1 Introduction

Autonomous agents that operate directly on a developer's local system introduce a category of risk that conversational AI does not. When an agent can delete files, execute commands, and reorganize directories, an ambiguous instruction like "clean this up" is no longer a safe experiment — it is a live operation on real infrastructure.

Clawed is a secure autonomous developer assistant built to demonstrate that autonomy and trustworthiness are not mutually exclusive. Rather than building a more capable agent, the focus is on building a more accountable one. Every action the system proposes is evaluated against two independent enforcement layers before a single byte is touched on disk. If either layer objects, the action is stopped, logged, and the reason is recorded.

The system targets a realistic and high-frequency developer scenario: routine project maintenance. Clawed can delete temporary files, run code formatters, reorganize directories, and generate commit messages — but only within boundaries the user explicitly defines.

2 System Architecture

Clawed is organized as a seven-layer pipeline. Each layer has a single, well-defined responsibility, and no layer bypasses the ones beneath it.

Layer Component Role
1Reasoning LayerInfers intent category from the user's natural-language prompt
2Intent ParserProduces a validated StructuredIntent schema object
3Action PlannerConverts the intent into a list of typed, atomic Action objects
4aDelegation GateConfirms the action type is within the sub-agent's granted scope (delegation path only)
4bLocal PolicyEngineValidates path bounds, file protections, and command allow/deny lists
4cArmorClaw VerifierIndependently confirms the action is semantically aligned with the declared intent
5OpenClaw ExecutorCarries out approved actions on the filesystem or via subprocess
6Audit LoggerRecords every stage decision to an immutable JSONL trace
Key Design Principle

The executor has no awareness of policy or intent. It receives only pre-approved action objects. Reasoning and execution are entirely separate modules that share no state.

3 Intent Model

When a user submits a prompt, the Reasoning Layer translates it into a named intent category using an OpenAI-compatible JSON adapter. When API credentials are absent, a deterministic keyword-matching fallback activates instead — ensuring the intent validation step is never silently skipped. The result is structured into a StructuredIntent dataclass:

{ "intent": "project_cleanup", "target_directory": "/home/user/project", "allowed_actions": ["delete_temp_files", "format_code"], "raw_prompt": "Clean the project and run formatting.", "reasoning_summary":"User asked to remove temporary artifacts..." }

The intent field is constrained to one of five named categories: project_cleanup, code_quality_maintenance, project_reorganization, commit_message_generation, and generic_project_operation. The planner uses the allowed_actions list — not the raw prompt — to generate the action plan, ensuring a clear semantic boundary between what the user said and what the system will attempt to do.

Not a hardcoded check

Intent is captured as a typed, validated schema and consumed by downstream components as structured data. The planner does not inspect the raw prompt. This ensures the enforcement chain operates on well-formed inputs, not free-form text.

4 Policy Model

The Policy dataclass encodes all runtime constraints as an immutable (frozen=True) configuration object. It is constructed once per run and passed to every enforcement component, eliminating any possibility of runtime modification.

{ "project_root": "./project", "allowed_directories": ["./project"], "protected_files": [".env", "credentials", "secret", "keys"], "protected_directories": ["./project/config", "./project/database"], "allowed_commands": ["python", "black", "ruff", "git"], "blocked_commands": ["sudo", "chmod", "curl", "wget"], "blocked_substrings": ["&&", ";", "|", "`", "$("], "max_command_runtime_seconds": 20 }

The policy covers the following constraint categories:

  • Directory-scoped access — all file paths must resolve within project_root and allowed_directories
  • Content restrictions — filenames containing tokens like .env or secret are blocked regardless of path
  • Command restrictions — executables are validated against both an allow-list and a deny-list
  • Injection prevention — shell metacharacters are scanned across the full command string
  • Time-based restriction — subprocess calls are terminated after a configurable timeout

5 Enforcement Mechanism

Every action produced by the planner must pass two independent enforcement gates before reaching the executor. Both must approve. Either can block independently, and the reason is always recorded.

Gate 1 — Local PolicyEngine

Performs structural safety checks: path canonicalization against project boundaries, protected file token matching, command allow/deny list evaluation, and shell metacharacter detection. Produces a PolicyDecision(allowed: bool, reason: str).

Gate 2 — ArmorClaw Verifier

Performs independent intent-alignment verification. When ARMORCLAW_API_KEY is configured, the action is POSTed to the ArmorClaw endpoint. Without credentials, a local intent-alignment check runs deterministically, verifying that the action type is semantically consistent with the declared intent. A DELETE action is valid for project_cleanup but not for commit_message_generation. Produces an ArmorClawDecision with a unique trace_id.

The two gates ask fundamentally different questions. The local policy asks: "Is this action structurally safe?" ArmorClaw asks: "Is this action semantically consistent with what the user asked for?" An action can be structurally safe (a valid path, a permitted command) while still being intent-misaligned. Both checks are necessary.

6 Bounded Delegation

Delegation Bonus Feature

Clawed implements bounded delegation through a DelegatedSubAgent, earning the bonus consideration outlined in the rulebook.

A parent agent can delegate a narrowly-scoped task to a DelegatedSubAgent by providing a DelegationScope object. The scope defines which action types the sub-agent may attempt, which commands it may run, and which directories it may access. At construction time, the scope is validated to be a strict subset of the parent's policy — it is structurally impossible to create a sub-agent with more authority than its parent.

scope = DelegationScope( allowed_action_types = ["run_command"], # only formatters allowed_commands = ["python", "black", "ruff"], delegated_by = "SecureDeveloperAgent", reason = "Format-only delegation" )

The sub-agent operates under a four-gate pipeline. The Delegation Gate fires first, before the local policy or ArmorClaw are consulted. Any action whose type is not in the granted scope is blocked immediately with a gate: "delegation" label. This means a sub-agent that plans a DELETE action will be stopped before any path checks or intent verification even begin — demonstrating that delegation enforcement is structurally independent from all other layers.

7 Observability and Traceability

Every decision in the pipeline is logged by the PipelineLogger to a JSONL audit trace. Each record carries a UTC timestamp, a stage name, and the full decision payload. The gate field on every execution result identifies exactly which enforcement layer made the final decision:

  • gate: "delegation" — stopped by the delegation scope check
  • gate: "local_policy" — stopped by the structural policy engine
  • gate: "armorclaw" — stopped by intent-alignment verification
  • gate: "all_passed" — approved by all gates and executed

This tagging makes it straightforward to audit any run: for each action, the trace shows what was proposed, which gate evaluated it, what the decision was, and what the executor produced.

8 Conclusion

Clawed demonstrates that intent-aware enforcement is not a theoretical concept — it is an engineering decision. By separating reasoning from planning, planning from enforcement, and enforcement from execution, the system ensures that no part of the pipeline can short-circuit the others. Ambiguous prompts do not reach the executor. Structurally valid but semantically misaligned actions are caught by ArmorClaw. Delegated sub-agents cannot exceed the authority they were given.

The result is an autonomous system that acts when it should, stops when it must, and always explains why.
